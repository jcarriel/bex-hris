Excelente análisis. Aquí está el resumen de lo que encontré y qué implica para tu integración:

---                                                                                                                                                                                                                          Lo que es UTimeMaster
                                                                                                                                                                                                                            
Es un sistema Django (Python) de ZKTeco que actúa como intermediario entre el dispositivo biométrico y tu base de datos. Expone una REST API en el puerto 8081 con Django REST Framework.

---
Cómo se comunica con el biométrico

Los dispositivos ZKTeco usan principalmente dos protocolos:

1. iClock/ADMS (el que usa UTimeMaster)

El dispositivo pushea activamente los registros al servidor. Funciona así:
Biométrico → HTTP POST → iclockservice (puerto configurable) → BD SQL Server
El dispositivo está configurado con una URL de servidor. Cada vez que alguien marca, envía los datos automáticamente.

2. SDK/TCP directo (lo que te interesa a ti)

Puedes conectarte directamente al dispositivo por TCP y pullear los datos:
Tu backend → TCP socket → Biométrico (puerto 4370 por defecto)

---
Las opciones que tienes

Opción A — Consumir la API de UTimeMaster (más fácil, ya hoy)

UTimeMaster ya expone todo en REST. Los endpoints clave que encontré:

┌──────────────────────────────┬─────────────────────────────────────────┐
│           Endpoint           │                 Qué da                  │
├──────────────────────────────┼─────────────────────────────────────────┤
│ GET /api/iclock/transaction/ │ Registros de marcación (lo que importa) │
├──────────────────────────────┼─────────────────────────────────────────┤
│ GET /api/iclock/terminal/    │ Dispositivos conectados                 │
├──────────────────────────────┼─────────────────────────────────────────┤
│ GET /api/att/payload/        │ Asistencia ya procesada                 │
├──────────────────────────────┼─────────────────────────────────────────┤
│ GET /api/att/manual-log/     │ Marcaciones manuales                    │
└──────────────────────────────┴─────────────────────────────────────────┘

Tu marcacionService podría hacer un GET /api/iclock/transaction/ con filtros de fecha en vez de depender de un upload manual.

Opción B — Conectarte directo al biométrico (sin UTimeMaster)

Para esto existen librerías open source que hablan el protocolo ZKTeco directamente:

Node.js (ideal para tu backend):
- node-zklib — la más usada, comunicación TCP al puerto 4370
- zkteco-device — alternativa más reciente

Lo que puedes hacer:
// Concepto - tu backend Node.js
import ZKLib from 'node-zklib'

const zk = new ZKLib('192.168.x.x', 4370, 10000, 4000)
await zk.createSocket()
const { data: attendances } = await zk.getAttendances()
// attendances: [{ deviceUserId, attTime, verifyMethod }]

---
Lo que necesitas saber del dispositivo físico

Para cualquiera de las dos opciones necesitas:
1. IP del dispositivo — está en la red local (revisa la pantalla del biométrico o el router)
2. Puerto — 4370 TCP por defecto en ZKTeco
3. Contraseña del dispositivo — usualmente 0000 o vacía si nunca fue cambiada
4. Modelo exacto — para saber qué comandos soporta (ZK100, ZK200, iClock series, etc.)

---
Mi recomendación para el corto plazo

Empieza con Opción A — consultar la API de UTimeMaster que ya está corriendo. Es la ruta de menor riesgo y cero configuración adicional. El endpoint GET /api/iclock/transaction/?start_time=FECHA&end_time=FECHA te daría 
las marcaciones directamente en tu backend, reemplazando el upload manual de Excel que tienes ahora.

Para el largo plazo, cuando quieras independizarte de UTimeMaster, implementamos un servicio Node.js con node-zklib que se conecta directo al dispositivo y sincroniza a tu BD SQLite.

 Basado en el protocolo TCP de ZKTeco (puerto 4370) y las librerías open source, estas son todas las acciones disponibles:

  └──────────────────────────────────┴───────────────────────────────────────────────────────┘

  ---
  Lo más valioso para tu HRIS

  El flujo ideal que podrías implementar sería:

  1. Cada X minutos → tu backend se conecta al biométrico
  2. Descarga marcaciones nuevas (desde última sincronización)
  3. Las guarda en tu BD SQLite
  4. Borra los registros del biométrico (para no reprocesar)
  5. Tu módulo de Asistencia los muestra en tiempo real

  Y en sentido inverso:
  1. Creas un empleado en tu HRIS
  2. Tu backend lo crea automáticamente en el biométrico
  3. El empleado puede marcar sin configuración manual

  ---
  Limitaciones importantes

  - Huellas y plantillas faciales — no se pueden leer/escribir por TCP en la mayoría de modelos (solo por USB con el SDK oficial). Solo puedes leer quién marcó y cuándo.
  - Modelo del dispositivo — no todos los ZKTeco soportan todos los comandos. Los modelos más antiguos tienen capacidades reducidas.
  - Un socket a la vez — el dispositivo solo acepta una conexión TCP simultánea. Si UTimeMaster ya está conectado, deberías desconectarlo o usar su API en su lugar.
  - Sin HTTPS — la comunicación es TCP plano, sin cifrado.
